# BEEIJA_PERMANENT_DEVELOPMENT_STANDARD

Last updated: July 2026

This is the locked permanent development standard for Beeija. Use this as the single source of truth for every future Beeija tool, UI update, shared component update, content update, SEO update, and production correction.

Beeija must not be treated as a generic calculator website. Beeija is a trust-first Cloud + AI cost planning website built to help users compare pricing, estimate usage, and make better decisions before building or buying.

---

## 1. Beeija Positioning

Beeija means: Seed → Beginning → Growth → Better outcomes.

Tagline:

**Seed today. Better outcomes tomorrow.**

Beeija’s role:

- Beeija helps users before building.
- Beeija helps with cost planning, usage planning, pricing comparison, forecasting, capacity planning, and decision support.
- Yoryantra helps users during building with developer utilities.
- Beeija and Yoryantra should feel like sibling websites created by the same individual.

Beeija tone:

- simple
- practical
- human
- creator-built
- trust-first
- clean
- focused
- useful
- not corporate
- not startup/SaaS-like
- not AI-template-like

Avoid language like:

- optimize your enterprise stack
- unlock cloud intelligence
- AI-powered cost platform
- mission-critical business solution
- transform your cloud operations
- maximize enterprise efficiency

Prefer language like:

- estimate monthly cost
- compare options before buying
- understand usage
- avoid surprise bills
- plan before choosing a provider
- copy current pricing from official pages
- use this as a planning estimate

Beeija’s long-term earning goal remains:

**USD $5,000–$50,000+ per month**

- useful tools

## 4. Tool Selection Standard

Every future Beeija tool should satisfy most of these:

- meaningful search volume
- relatively low competition
- strong commercial intent
- high CPC potential
- practical user value
- evergreen usefulness
- uncommon or under-served topic
- clear cost, pricing, capacity, usage, comparison, or planning angle
- useful before purchase, deployment, or provider selection
- relevant to cloud, AI, hosting, infrastructure, API, SaaS, capacity, or technical decision support

Avoid:

- random calculators
- toy tools
- generic utility-site filler
- low-intent pages
- simple math calculators with no commercial/planning value
- AI gimmicks
- duplicate tools with minor wording changes
- tools that do not help a real user make a decision

## 5. SEO Standard

Beeija is SEO-first but not SEO-spam.

Each tool should target real search intent naturally.

Metadata must be:

- clean
- practical
- specific
- human
- relevant to the tool

Avoid metadata phrases like:

- Online Free
- Free Online
- instantly
- fastest
- best ever
- ultimate
- AI-powered
- enterprise solution

Do not keyword-stuff.

Do not write pages that sound like SEO filler.

Each tool should include helpful, natural sections where relevant:

- tool-specific intro
- calculator UI
- estimate assumptions
- what the result includes
- official pricing sources
- when the calculator helps
- how to read the result
- common questions
- automatic related tools

Do not force every tool into exactly the same section structure if the tool does not need it.

## 6. Content Standard

Every Beeija tool must have natural, tool-specific content.

Do not repeat the same AI-template wording across tools.

Avoid repeated generic phrases like:

- This tool helps you optimize your workflow.
- Make smarter business decisions.
- Perfect for startups, developers, and teams.
- Improve your cloud strategy.
- Unlock cost intelligence.
- Save time and money with this powerful tool.

Instead, write content that matches the exact tool.

Examples:

OpenAI cost tool should talk about:

- input tokens
- output tokens
- model choice
- cached tokens
- prompt volume
- monthly API planning

Cloud NAT tool should talk about:

- gateway hours
- processed data
- public IP cost
- cross-zone transfer
- flow logs
- regional differences

Cloud Load Balancer tool should talk about:

- hourly runtime
- capacity units / LCU usage
- processed data
- forwarding rules
- outbound transfer
- WAF
- logging
- setup or migration cost

No copied feature blocks.

No filler.

No content-farm tone.

Official pricing verification is mandatory for every pricing tool.

Examples:

- Azure pricing pages

- Cloudflare pricing pages

- OpenAI pricing pages
- Anthropic pricing pages
- Google Gemini pricing pages

- Grok/xAI pricing pages

## 8. Pricing Checked Date Rule

Every Beeija pricing page should show a pricing checked date.

Example:

**Pricing model checked: July 7, 2026**

or

**Pricing checked: July 2026**

Use exact date where possible.

The date should be visible to users and should be bold where it appears in pricing-source context.

---

## 9. Estimate and Billing Disclaimer Rule

Beeija tools must never claim exact billing truth.

Avoid:

- exact bill
- guaranteed monthly bill
- final invoice amount
- official quote
- precise cost
- complete billing estimate

Use:

- planning estimate
- monthly planning cost
- rough monthly estimate
- before taxes
- before discounts
- before credits
- before enterprise agreements
- before provider-specific billing rules
- verify in the provider calculator before purchase

Every pricing result should explain:

- what was counted
- what was excluded
- what may change the real bill
- what the user should check before buying/deploying

---

## 10. Custom Pricing Rule

Custom/editable pricing inputs are important for Beeija.

Provider pricing often changes by:

- region
- currency
- service tier
- account agreement
- usage unit
- discount
- commitment
- free tier
- tax
- enterprise contract

When pricing is complex or changes often, keep rates editable.

Do not pretend one hardcoded rate is always true.

Blank price fields are acceptable when safer.

If rates are blank, explain why:

> Prices are blank because provider rates vary by region, product type, account agreement, currency, and discounts. Enter the current numbers from the official pricing pages.

## 12. Central Control Rule

This rule is locked.

When the user says:

- central
- centrally controlled
- shared
- shared component
- apply across all tools

it means fully centrally controlled, not half-centralized.

If required files are missing, ask for them first.

Do not generate incomplete production files.

Do not claim something is centrally controlled if only one page was changed.

For true central changes, inspect and update the required files, such as:

- `app/globals.css`
- shared components
- relevant tool page/client
- `layout.tsx` if title/metadata/global shell is involved
- reference tool files if matching an existing style
- `tools.ts` only if the tool data itself must change

No half-baked solutions.

This is live production.

## 15. Typography Standard

Main page text should be readable, especially for users wearing eyeglasses.

Calculator controls should be compact enough for dense input grids.

Balance:

- page intro/body: readable
- section copy: readable
- input labels/helper text: slightly smaller
- input values: compact but clear
- result totals: prominent
- footer/listing navigation: can stay smaller

Do not make the calculator grid feel bulky.

Do not make helper text tiny.

Spacing should feel like the approved NAT Gateway / Load Balancer rhythm.

Avoid:

- sections stuck together like one paragraph

- excessive card stacking
- cramped FAQ/related tools area

Use consistent spacing across all content sections:

- feature card sections
- content sections
- official pricing sources
- common questions
- related tools

Spacing must be centrally controlled where possible.

## 19. Input Field Standard

Inputs should follow the approved clean Google/AWS-style pattern.

Rules:

- compact height
- readable labels
- helper text below
- prefix/suffix inside the field
- no overly segmented borders
- no oversized values
- clean focus state
- layout must not break with long numbers

Use shared `BeeijaNumberField`.

---

## 20. Long Number Rule

Beeija tools must not break layout when users enter very long numbers.

Rules:

- protect inputs from absurdly long values where possible
- sanitize numeric input
- truncate or contain long displayed results where needed
- use comma-formatted full numbers
- do not wrap huge values into ugly multi-line blocks
- show practical warning/help text for unusually large planning values

Stress-test with long numbers before shipping.

---

## 21. Number Formatting Rule

Do not use K/M/B/T shorthand in pricing tools.

Use full numbers with commas.

Good:

- `$1,250.00`
- `$125,000.00`
- `1,000 GB`

Avoid:

- `$1.2K`
- `$3.4M`
- `$8B`
- `$2T`

For extremely unrealistic values, use safe wording such as:

- Very high estimate
- Use a smaller planning value or split the workload into parts

## 23. Common Questions Rule

Common Questions should be visible.

Do not use dropdown arrows.

Do not make users click to read FAQ answers.

Question count is not fixed.

Usually:

- 3 questions for simpler tools
- 4–5 questions for more complex tools
- more only if genuinely useful

Questions must be natural and tool-specific.

Do not add filler questions to make the page longer.

Use a simple yellow vertical line list where it improves readability.

- “When This Calculator Helps”

- important usage notes

## 25. Provider Comparison Rule

Provider comparison should be simple and information-focused.

Do not use decorative green number circles like 1, 2, 3, 4 unless there is a real ranking need.

For comparison lists, show:

- provider name
- service/product name
- estimated cost
- selected state
- clean hover/focus state

## 26. Related Tools Rule

Related tools must stay automatic.

Do not hardcode related links manually.

Use current href:

```tsx
<BeeijaRelatedTools currentHref="/tools/tool-slug" />
```

or the approved section wrapper:

```tsx
<BeeijaRelatedToolsSection currentHref="/tools/tool-slug" />
```

The related tools should come from `tools.ts`.

Related section heading should be natural by category, for example:

**Explore Related Cloud Cost Tools**

Do not manually pass unsupported props like `category` unless the component explicitly supports it.

---

## 27. `tools.ts` Rule

Keep `tools.ts` small.

Standard entry:

```ts
{
  title: "Cloud Load Balancer Cost Comparison Calculator",
  description:
    "Compare AWS, Azure, Google Cloud, and Cloudflare load balancer costs with editable hourly, capacity, traffic, WAF, logging, transfer, and setup inputs.",
  href: "/tools/cloud-load-balancer-cost-comparison-calculator",
  category: "Cloud Cost Calculators",
},
```

Do not add `keywords` by default.

Only use extra fields like `keywords` if there is a valid functional reason, such as:

- internal search ranking
- alias matching
- typo matching
- related-tool scoring
- tag filtering

If keywords are needed, explain the reason first.

SEO keywords belong in page metadata, not normally in `tools.ts`.

Do not change hrefs.

Do not change categories unless intentionally restructuring.

Do not change titles unless clearly needed.

---

## 28. Metadata Rule

Tool metadata must be clean and title-safe.

If root layout already applies a title template such as `| Beeija`, page metadata should not manually duplicate it.

Correct:

```ts
title: "Cloud Load Balancer Cost Comparison Calculator"
```

Avoid:

```ts
title: "Cloud Load Balancer Cost Comparison Calculator | Beeija"
```

if layout already appends `| Beeija`.

Open Graph and Twitter titles may include `| Beeija` if appropriate for sharing previews.

Always check browser title after deployment.

---

## 29. Breadcrumb and Back Link Rule

Do not duplicate breadcrumbs.

If a layout already renders:

`Home / Tools`

the tool page/header should not render it again.

Keep:

`← Back to Tools`

Back link must include the arrow.

Tool category should appear below Back to Tools and use Beeija/Yoryantra-style yellow uppercase label.

## 30. Page Title Rule

Use the larger Yoryantra/Beeija-style tool page title.

The main title should feel strong and clear for users and search engines.

Compact titles can be used inside cards or result panels, not the main hero.

- inputs

- plan cards

## 32. Production Safety Rule

Do not touch unrelated files.

Do not randomly redesign.

Do not update all tools at once.

Only replace files needed for the specific change.

When packaging files, include correct repo paths.

Avoid overwriting unchanged production files.

---

## 33. Batch Workflow Rule

Use controlled batches.

For polishing existing live tools:

- one tool at a time when applying a new central UI standard
- or maximum 3 tools per batch after the pattern is proven

For new tool development:

- 3 tools per batch only when the shared pattern is stable
- otherwise finish one reference tool first

Reason:

- avoids breaking 192 tools
- easier to check logic
- easier to check UI
- easier to run build
- easier to deploy safely

---

## 34. Completed Tools / Skip List Rule

Maintain a completed-tools skip list.

Do not suggest or rework already completed tools unless there is a real reason.

Do not repeat tools by mistake.

Once a tool is updated to the approved central Beeija standard, mark it as completed for that migration pass.

## 35. Build and TypeScript Rule

Before returning production files where possible:

- check TSX syntax
- check imports
- check JSX validity
- avoid unsupported props
- avoid missing component imports
- run available TypeScript/transpile checks

After replacing files, user should run:

```bash
npm run build
```

A production update is not complete until the build passes.

## 39. Migration Rule for Existing Tools

When applying the new Load Balancer standard to existing tools:

1. Pick one live tool.
2. Compare against Load Balancer approved reference.
3. Move only reusable improvements into shared components.
4. Update that tool to use shared components.
5. Keep `tools.ts` untouched unless needed.
6. Run build.
7. Deploy and visually check.
8. Mark the tool complete.
9. Move to the next tool.

Do not update all live tools blindly.
